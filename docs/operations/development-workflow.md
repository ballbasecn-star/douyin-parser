# 开发与提交流程

## 目的

本文档定义项目的日常开发流程、最小验证要求，以及 Git 提交规范。

它面向人和 AI 共同适用。  
其中更偏“AI 执行动作”的规则写在 `AGENTS.md`，本文档则作为团队级工程流程规范。

## 基本原则

### 1. 小步提交

一次 commit 应只表达一个逻辑完整的改动，例如：

- 一个功能
- 一个 bug 修复
- 一次文档补充
- 一次可独立理解的重构

避免把多个无关改动混在同一个 commit 中。

### 2. 先验证，再提交

提交前应尽量完成适用于当前改动的最小验证。  
如果无法验证，也必须在提交说明或工作汇报中明确说明。

### 3. 代码与文档同步演进

当改动会影响以下内容时，应同步更新文档：

- 产品范围
- API 契约
- 数据结构
- 部署方式
- 技术决策
- 模块职责

## 标准开发流程

建议采用以下顺序：

1. 阅读当前路线图与相关模块文档
2. 明确任务边界
3. 实现改动
4. 执行最小必要验证
5. 更新相关文档
6. 执行 `git commit`
7. 视情况再执行 `git push`

## 提交前最小验证

按改动类型执行适用项：

### 代码变更

至少应做以下一项或多项：

- 语法检查
- 单元测试
- 关键接口调用验证
- 本地手动功能验证

### 文档变更

至少应检查：

- 文档是否与当前代码和路线图一致
- 文档入口是否已被纳入 `AGENTS.md` 或相关索引

### 部署变更

至少应检查：

- 配置文件语法
- compose 或容器配置是否自洽
- 关键环境变量是否有说明

## Commit 粒度规范

适合单独成 commit 的例子：

- 新增 `/api/parse-sync`
- 新增博主监控 V1 PD
- 引入 PostgreSQL 相关建模
- 修复主页链接解析 bug

不适合混在一起的例子：

- 功能开发 + 无关样式调整
- API 改动 + 完全独立的文档重写
- 一个功能 + 另一个未完成功能的半成品

## Commit Message 规范

推荐使用以下前缀：

- `feat:`
- `fix:`
- `docs:`
- `refactor:`
- `chore:`

### 示例

- `feat: 添加同步解析接口`
- `fix: 修复文案分析返回解析失败`
- `docs: 补充环境策略与部署说明`
- `refactor: 拆分博主同步服务`

### 要求

- 标题简洁明确
- 直接写结果，不写过程口号
- 不使用模糊标题，如：
  - `update`
  - `misc`
  - `final`
  - `改一下`

## Push 规则

### 默认规则

- `commit` 可以在本地完成
- `push` 默认不是自动动作
- 只有在用户明确要求时，才执行 `git push`

### 生产相关规则

如果改动涉及：

- 生产部署
- 环境变量
- nginx 配置
- 数据库结构

则在 `push` 前应确保：

- 文档已更新
- 最小验证已完成
- 变更结果可描述

## 分支建议

当前仓库如果继续采用主分支直接迭代，至少要保证 commit 粒度清晰。  
如果后续协作人数增加，建议增加分支策略并单独补充文档。

## 何时拆多个 commit

以下情况应拆开提交：

- 功能实现和文档补充相互独立
- 数据库结构改动和业务逻辑改动可独立理解
- 一个任务内存在两个可回滚单元

## 何时更新本文档

当以下规则变化时应更新：

- 提交粒度要求
- 验证要求
- Commit Message 规范
- Push 流程
- 分支策略
