# 博主监控 V1 功能需求文档

## 摘要

博主监控 V1 会把 `douyin-parser` 从单视频解析工具扩展成一个以博主为中心的工作流。

第一版只聚焦于：

1. 维护一组对标博主
2. 将主页分享短链解析为稳定博主标识
3. 同步每个博主的视频列表及基础数据
4. 允许用户手动选择某些视频做转录与 AI 分析

这一版有意不做成全自动系统。

## 用户问题

现在用户研究同行博主时，一般是在抖音里手工浏览，再把某些视频链接一条条复制进单视频解析器。

这种方式会带来几个问题：

- 对标博主不能沉淀为长期资产
- 有价值的视频很容易丢失
- 视频数据不能积累成结构化素材库
- 转录和分析过早开始，在用户还没决定哪些视频值得深挖之前就已经消耗资源

## 产品目标

建立这样一套工作流：

1. 博主只需添加一次
2. 博主持久保存在管理列表中
3. 该博主的视频会被同步进结构化库
4. 用户选择值得深挖的视频
5. 分析结果可以长期保存，供后续生成与策划使用

## 目标用户

- 观察同领域对标账号的抖音创作者
- 搭建同行素材库的内容运营
- 基于同行视频做选题研究的小团队

## 核心流程

### 流程 1：添加博主

1. 用户粘贴一个博主主页分享短链
2. 系统解析短链并提取稳定博主标识
3. 系统保存博主记录
4. 系统可选地进行一次最近视频的初始同步

### 流程 2：查看博主视频

1. 用户进入某个博主详情页
2. 系统展示该博主已同步的视频列表和基础指标
3. 用户浏览、排序并识别值得分析的视频

### 流程 3：分析选中视频

1. 用户点击某条视频的“分析”
2. 系统调用已有的单视频解析/转录/分析流程
3. 系统保存转录文本与 AI 分析结果
4. 用户后续可以再次查看

## 功能范围

### A. 博主管理

系统必须支持：

- 输入主页分享短链
- 解析出稳定博主标识
- 保存博主元数据
- 自动防止重复添加
- 支持更新博主标签、备注或状态

建议保存字段：

- `source_url`
- `resolved_url`
- `stable_user_id`
- `nickname`
- `display_handle`
- `avatar_url`
- `domain_tag`
- `remark`
- `status`
- `created_at`
- `updated_at`

### B. 博主视频同步

系统必须支持：

- 拉取博主视频列表
- 处理分页
- 保存每条视频的基础信息与指标
- 对已有视频更新而不是重复插入
- 记录同步时间

建议保存字段：

- `creator_id`
- `video_id`
- `title`
- `description`
- `share_url`
- `cover_url`
- `publish_time`
- `play_count`
- `like_count`
- `comment_count`
- `share_count`
- `collect_count`
- `first_seen_at`
- `last_synced_at`

### C. 手动深度分析

系统必须支持：

- 由用户手动触发某条视频的转录与 AI 分析
- 复用已有的单视频解析内核
- 持久化保存转录文本和分析结果
- 在失败时展示状态与错误信息

建议保存字段：

- `video_id`
- `transcript`
- `analysis_json`
- `transcript_provider`
- `analysis_model`
- `status`
- `error_message`
- `created_at`
- `updated_at`

## V1 非目标

V1 不包含：

- 自动分析所有博主视频
- 定时按天或按小时自动同步
- 自动报告生成
- 主题聚类
- 大规模历史回溯
- 完整增长趋势分析

## 页面形态

### 博主列表页

- 添加博主输入框
- 博主卡片或表格
- 最近同步时间
- 状态
- 进入详情页入口

### 博主详情页

- 博主信息头部
- 已同步的视频列表
- 按时间或互动指标排序/筛选
- 对单条视频执行分析的入口

### 视频分析详情

- 视频基础信息
- 转录文本
- AI 分析结果
- 当前分析状态与错误状态

## API 方向

建议新增如下接口：

- `POST /api/creators`
- `GET /api/creators`
- `GET /api/creators/:id`
- `PATCH /api/creators/:id`
- `POST /api/creators/:id/sync`
- `GET /api/creators/:id/videos`
- `POST /api/videos/:id/analyze`
- `GET /api/videos/:id/analysis`

返回格式建议与当前同步 API 保持一致：

- `success`
- `data`
- `error`

## 验收标准

当满足以下条件时，V1 可以认为达标：

1. 用户可以通过主页分享链接添加一个博主
2. 系统可以保存稳定的博主标识
3. 系统可以同步并展示该博主最近的视频列表
4. 用户可以选择一条视频并触发转录与 AI 分析
5. 结果可以持久化并再次打开查看
6. 博主和视频记录都能自动去重

## 风险

- 博主主页接口可能比单视频详情接口更不稳定
- Cookie 时效依然是关键依赖
- 一旦取消“手动选择”边界，分析成本会快速上升
- 在开始累积数据之前，数据结构需要先定稳

## 发布原则

先把“博主”和“视频记录”做成稳定的持久化资产，再考虑自动化。只有在视频列表同步和手动分析都稳定后，才考虑继续往自动调度扩展。
