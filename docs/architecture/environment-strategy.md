# 环境策略

## 目的

本文档定义项目的运行环境划分、各环境职责、配置隔离原则，以及从开发到生产的发布流向。

## 当前结论

当前阶段，项目采用两套主要环境：

- `dev`：本地开发与功能验证环境
- `prod`：正式生产环境

当前不单独引入 `staging`。如果后续发布复杂度显著上升，再通过新的 ADR 评估是否新增。

## 环境定义

### dev

用途：

- 本地开发
- 功能联调
- 文档与接口验证
- Docker 镜像本地构建与测试
- 数据库 schema 迭代验证

特点：

- 允许更高频变化
- 可以使用测试数据或局部真实数据
- 允许调试日志更详细
- 可以容忍人工操作较多

### prod

用途：

- 对外提供正式服务
- 承载真实 Cookie、真实 API Key、真实业务数据
- 作为长期运行的正式环境

特点：

- 追求稳定和可恢复
- 配置、数据和证书必须持久化
- 所有变更应可追踪、可回滚

## 环境目标差异

| 维度 | dev | prod |
|------|-----|------|
| 主要目的 | 开发与验证 | 对外提供正式服务 |
| 数据要求 | 可使用测试或局部真实数据 | 使用真实持久化业务数据 |
| 变更频率 | 高 | 低 |
| 日志级别 | 可偏详细 | 以可观测与稳健为主 |
| 发布方式 | 本地构建与手动验证 | 受控发布 |
| 容错要求 | 较低 | 高 |

## 配置隔离原则

不同环境必须隔离以下内容：

- 环境变量
- 数据库
- Cookie 数据目录
- 证书
- 域名入口
- 镜像 tag

### 环境变量

建议：

- `dev` 使用本地 `.env.dev` 或等价方式
- `prod` 使用服务器上的 `.env`

禁止将生产环境的真实 API Key 和生产 Cookie 复制到本地仓库。

### 数据库

建议：

- `dev` 使用独立开发数据库
- `prod` 使用正式生产数据库

禁止开发环境直接连接生产数据库。

### 文件数据

Cookie、导出文件、缓存文件等都必须按环境隔离目录。

例如：

- `dev`: 本地项目目录或本机专用目录
- `prod`: 服务器上的持久化挂载目录

## 部署模型

### dev

推荐方式：

- 本地 Python 直接运行，适合快速调试
- 本地 Docker 容器运行，适合验证镜像和依赖

### prod

推荐方式：

- Docker 容器运行
- 共享 `nginx` 负责统一入口与 TLS
- 数据目录和配置目录通过宿主机挂载持久化

## 数据隔离原则

任何环境都不应默认共享以下资源：

- PostgreSQL 数据库实例中的同一业务库
- Cookie 文件目录
- `.env` 文件
- 日志目录

即使使用同一台服务器，也应通过：

- 独立数据库名
- 独立 schema
- 独立宿主机目录
- 独立容器名或 compose 项目名

实现隔离。

## 镜像与版本策略

建议：

- `dev` 使用开发 tag，如 `dev` 或 `sha-<commit>`
- `prod` 使用固定版本 tag，如 `vX.Y.Z`

禁止在 `prod` 长期直接使用不受控的 `latest` 作为唯一部署依据。

## 发布流向

推荐流向如下：

1. 在 `dev` 完成功能开发
2. 在 `dev` 完成本地验证和镜像验证
3. 生成固定版本镜像
4. 将固定版本部署到 `prod`
5. 在 `prod` 验证健康检查、关键 API 和反向代理

## 当前阶段不做的事

本文档明确当前阶段不做：

- 单独的 `staging` 环境
- 自动化灰度发布
- 多地域环境
- 多租户环境隔离

## 何时更新本文档

当以下内容变化时必须更新：

- 新增或移除环境
- 发布流程变化
- 配置隔离方式变化
- 引入 `staging`
- 引入多数据库或多服务部署策略
