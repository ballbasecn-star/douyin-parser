# ADR 0002：博主监控阶段采用 PostgreSQL 作为主数据库

## 状态

已接受

## 背景

项目从“单视频解析工具”向“博主监控系统”演进后，需要长期保存以下数据：

- 对标博主
- 博主视频列表
- 视频基础指标
- 视频转录结果
- 视频 AI 分析结果

原有的文件存储和前端本地存储只能支撑工具型使用，无法可靠支持：

- 去重
- 列表查询
- 排序筛选
- 后续扩展

因此需要正式引入关系型数据库。

## 备选方案

### 方案 A：SQLite

优点：

- 接入简单
- 本地开发方便

缺点：

- 更适合单机原型
- 不适合作为长期线上主存储
- 后续扩展到更复杂查询和并发场景时迁移成本更高

### 方案 B：MySQL

优点：

- 常见、成熟

缺点：

- 对当前项目没有明显优于 PostgreSQL 的收益
- 后续涉及 JSON 数据和复杂查询时，PostgreSQL 更自然

### 方案 C：PostgreSQL

优点：

- 适合作为长期线上主存储
- 关系建模能力强
- JSON / JSONB 支持好
- 适合后续做更复杂的筛选、聚合和趋势分析
- 与 Python 生态集成成熟

缺点：

- 比 SQLite 更重
- 需要单独部署和迁移管理

## 决策

从“博主监控 V1”开始，项目的主业务数据库采用 PostgreSQL。

第一版范围内，以下核心数据必须进入 PostgreSQL：

- creators
- creator_videos
- video_analyses

以下内容不作为 PostgreSQL 第一版核心数据：

- Cookie 文件
- 浏览器本地历史记录
- API Key

## 结果影响

### 正面影响

- 为博主监控建立稳定的数据基础
- 可以可靠做去重、查询、筛选和持久化
- 为后续快照、趋势分析、任务调度打基础
- 减少后续从轻量存储迁移的返工

### 负面影响

- 引入数据库部署和迁移复杂度
- 需要新增 repository、ORM 和迁移层

## 配套要求

为配合本决策，后续实现应同时满足：

1. 数据结构通过迁移工具管理
2. 不把数据库访问直接写进路由层
3. 新增持久化逻辑通过 repository 层封装
4. 新功能优先围绕 PostgreSQL 建模，而不是先写文件存储再迁移

## 何时重新评估

只有在以下情况出现时，才重新评估本决策：

- 项目重新收缩为纯本地单机工具
- 监控能力被完全取消
- 部署环境对 PostgreSQL 存在不可接受的限制
